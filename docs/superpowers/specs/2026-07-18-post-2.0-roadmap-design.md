# Post-2.0 Roadmap Design

> 状态: Approved (2026-07-18)
> 范围: 2.0.1 → 2.1 → 2.2 → 2.3 → 3.0
> 取向: 先还债再前进（用户决策）
> 交付粒度: 版本级里程碑

---

## 0. 背景与诊断

2.0.0 已发布（13 格式 + WASM + Pipeline）。基线: build OK, 287 tests pass（137 unit + 150 integration）。

全面审查发现的债务分布（详见 `docs/comprehensive-review-2026-07-16.md` 同期产出）:

- **P0 发布工程瑕疵**（4 项）: clippy 失败、CHANGELOG 无 `[2.0.0]` 头、Cargo.toml "12 formats"、README 文档已删字段。
- **P1 正确性 bug**（6 项）: EBU STL round-trip 损坏、SCC drop-frame 忽略、encoding UTF-16 不剥 BOM、parse_to_file 对二进制格式误解码、main.rs unwrap 脆弱、split_long 产零时长。
- **P2 API 一致性**（6 项）: 13 格式公共 API 不统一（generate/parse_stream/write_stream/返回类型/ttml header）。
- **P3 测试与文档**（8 项）: WASM 零覆盖、3 格式无 round-trip、cross-format 仅 3% 覆盖、proptest ASCII-only、CI 不测 WASM/MSRV。

诊断画像: **"广度优先"扩张期的典型债务**。无结构性烂账，但深度跟不上广度。2.x 用 2-3 个版本还债，3.0 综合 4 个前进方向。

## 0.1 用户决策（已敲定）

| 决策点 | 选择 |
|--------|------|
| 版本取向 | 先还债再前进 |
| SCC + EBU STL 正确性 | 修对它（含 drop-frame 算法 + 二进制 round-trip） |
| WASM 策略 | 补到一等公民（CI + 测试 + npm） |
| 3.0 前进方向 | 全选: 深挖变体 + AI 集成 + zero-copy + 跨语言绑定 |

## 1. 整体结构与发布节奏

```
2.0.1   2.1      2.2      2.3      2.4       3.0          3.1
─────   ──────   ──────   ──────   ──────    ──────────   ──────────
hotfix  正确性   API拉齐  测试/CI  gap收编   综合前进     性能深化
✅      ✅       ✅       ✅       DFXP      AI翻译       zero-copy
                                   Whisper   npm/PyO3/C   SMPTE-TT
                                   去重      IMSC         Spruce STL
                                   normalize CEA-708     Node native
1-2天   3-4周    3-4周    2-3周    ~1周      6-8周        4-6周
patch   patch    minor    patch    minor     major        minor/major
```

> **v2.4 + v3.0 + v3.1 是原 v3.0 拆分而来**（基于 `docs/subtitler-vs-editingtools-gap-analysis.md` 的发现）。
> 拆分理由: (1) gap analysis 的 P1 项（DFXP/Whisper/去重）低投入高回报, 不该等 3.0;
> (2) zero-copy 改 Subtitle 结构, 与 3.0 绑定"稳定 API"目标冲突, 独立成 3.1。

### 1.1 贯穿原则

1. **每个 bug 修复配套 regression 测试**——不"先修后补测"。
2. **每版本单一主题**——便于 review、release notes、回滚。
3. **2.0.1 / 2.3 是零风险版本**（纯文档/纯测试），作为节奏阀门吸收延期压力。
4. **API 破坏性变更只在 2.2 和 3.0**；其余严格向后兼容。
5. **每个版本独立 release 分支**（`release/2.1.x` 等）；2.0.1 直接在 `main` 打 tag。

### 1.2 版本号策略

SemVer: `patch` = 修复/文档；`minor` = 新 API 或 minor breaking（配 MIGRATION 说明）；`major` = 重大架构或大量新 API 面。

### 2.2 — v2.1 (正确性债务)

**目标**: 修复 6 项 P1 正确性 bug。**每项修复配套 regression 测试**。无新公共 API。

| # | 文件 | bug | 修复方案 | 测试 |
|---|------|-----|---------|------|
| C1 | `src/ebu_stl.rs:392-393` | `to_string` 用 `sub.start / 40` 魔法数; `encode_smpte_timecode` 又把它当毫秒重编码 → round-trip 损坏 | 存毫秒直接，与 `parse_smpte_timecode` 输出契约对齐; `model/builder.rs:183-184` 同步修 | 新增 `test_ebu_stl_round_trip`: parse → to_string → parse 后 start/end 误差 < 40ms |
| C2 | `src/scc.rs:172-176, 179-200` | drop-frame timecode (`;`) 用非 drop-frame 公式, NTSC 每小时漂 3.6s | 实现 SMPTE 12M drop-frame 算法（每分钟丢 2 帧, 每 10 分钟不丢）; `ms_to_scc_timecode` 同步 | 新增 `test_scc_dropframe_accuracy`: 已知时码对照表; `test_scc_round_trip` |
| C3 | `src/encoding.rs:33-52` | UTF-16BE/LE 不剥 U+FEFF BOM（与 UTF-8 路径不一致）; `chunks_exact(2)` 静默丢奇字节 | 跳过 2 字节 BOM 后再 chunks; 结果 `trim_start_matches('\u{FEFF}')`; 短输入守卫 | 新增 `test_decode_utf16_bom_stripped`, `test_decode_utf16_odd_byte` |
| C4 | `src/main.rs:96-126, 135` | `parse_to_file` / `cmd_parse` 对所有输入先跑 `decode_to_string`, EbuStl 二进制文件触发误报 InvalidEncoding | 把 `decode_to_string` 移进非二进制分支; EbuStl 跳过解码 | 新增 `test_parse_binary_stl_not_decoded` |
| C5 | `src/main.rs:386-389` | `durations.iter().min().unwrap()` 依赖上游早返回的隐式不变量 | 改 `.copied().unwrap_or(0)` 把安全性局部化 | 现有测试足够（非回归） |
| C6 | `src/model/trait.rs:245` | `split_long` 整数除法; `num_chunks > duration` 时产零时长 | `div_ceil` 分配余数; `if end <= start { continue }` 守卫 | 新增 `test_split_long_no_zero_duration` |

**关键决策点（C2 SCC drop-frame）**: drop-frame 是 NTSC 的硬性数学, 标准公式:

```
total_frames = 30*60*60*h + 30*60*m + 30*s + frames   // 用 30 不是 29.97
total_frames -= 2 * (total_minutes - total_minutes / 10)  // 每分钟丢 2, 每 10 分钟不丢
ms = total_frames * 1000 / 29.97
```

实施时加注释引用 SMPTE 12M-1-2014 §3.3。

**验收**:
- 全部新增 regression 测试通过。
- `cargo test --all-targets` 全绿。
- 现有 SCC/EBU STL 用例不回归（若有依赖错误行为的, 一并修正）。
- clippy 零警告。

**发布**: `release/2.1.x` 分支, tag `v2.1.0`。

### 2.3 — v2.2 (API 拉齐)

**目标**: 13 格式的公共 API 表面统一。**含 minor breaking change**（返回类型统一）, 配 `MIGRATION.md` 说明。

| # | 现状 | 目标 | breaking? |
|---|------|------|-----------|
| A1 | 仅 `srt`/`vtt` 有 `generate(subs, path, policy)` | 13 格式全补 `generate` | 否 |
| A2 | 仅 `srt`/`vtt` 暴露 `parse_stream(content)` 构造器 | 9 个有 StreamingParser impl 的格式全暴露（srt/vtt/ass/ttml/sami/scc/lrc/microdvd/subviewer/sbv/mpl2） | 否 |
| A3 | TTML `write_stream` 用同步 `std::io::Write` | 改为 `tokio::io::AsyncWrite + Unpin`（其余已是） | 否（仅 TTML 用户） |
| A4 | 返回类型分裂: `AnyResult<SubtitleFile>` vs `Result<_, SubtitleError>` | 统一为 `AnyResult<SubtitleFile>`（对外）/ `SubtitleError`（内部）; 跨格式用 `?` 不撞类型 | **minor breaking** |
| A5 | `ttml::to_string(subs, _header)` 第二参数被静默丢弃 | 实现真正保留 header, 或删该参数 | minor breaking |
| A6 | MicroDVD 把 `fps` 贯穿所有签名, 破坏一致性 | 保留（功能必需）, 但用 builder 风格 `with_fps` 收敛, 函数签名用 `Option<f64>` | 否 |

**实施顺序**（先非 breaking, 后 breaking）:
1. A1/A2/A3/A6（纯新增/重构, 不影响现有调用）。
2. A4 返回类型统一（一次性, 配 `#[deprecated]` 过渡期 1 版本）。
3. A5 TTML header。

**验收**:
- 13 格式 API 表面通过 `tests/api_surface.rs`（新增）参数化测试, 验证每个格式都暴露统一签名。

## 3. 依赖与风险

### 3.1 跨版本依赖

```
2.0.1 ──→ 2.1 ──→ 2.2 ──→ 2.3 ──→ 2.4 ──→ 3.0 ──→ 3.1
         (正确性) (API)   (锁死)  (gap)   (前进) (性能)
                  ✅       ✅      ✅
```

- 2.1 必须先于 2.2: API 拉齐若基于错误实现, 拉了也白拉。
- 2.3 必须跟在 2.2 后: 测试锁刚改的 API 才有意义。
- 2.4 是 gap analysis P1 的低投入收编, 不依赖任何前置版本。
- 3.0 依赖 2.x 全部完成: AI 集成、绑定、新格式都建立在稳定 API 之上。
- 3.1 包含 zero-copy（breaking），必须在 3.0 绑定 API 稳定后才动。

## 5. 不做什么（YAGNI 边界）

明确排除, 避免范围蔓延:

- ❌ **不做 GUI/Web 前端**: CLI + 库 + 绑定已够, 前端交由下游。
- ❌ **不做视频嵌入/时间轴编辑**: 这是播放器/NLE 的职责。
- ❌ **不做字幕渲染/字体**: libass 已是事实标准。
- ❌ **不支持 P2P/网络协议**: 不做字幕同步分享。
- ❌ **2.x 不引入 async-trait / GAT 重构**: 保持现状, 3.0 视需要。
- ❌ **不做插件系统**: YAGNI, 现有 feature flags 够用。
- ❌ **3.0 不重写 error 类型**: thiserror 枚举已够, 把重构留到 4.0。
