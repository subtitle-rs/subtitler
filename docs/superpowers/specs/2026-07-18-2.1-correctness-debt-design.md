## 2. 技术方案

### 2.1 Fix 1: EBU STL round-trip

**根因**: `to_string` (ebu_stl.rs:392) 用 `(sub.start / 40) as u32` 把毫秒转"未知单位"塞进 `TtiBlock.timecode_start`, 但 `TtiBlock::to_bytes` → `encode_smpte_timecode` 把这个值当**毫秒**重编码成 SMPTE 帧格式。parse 路径 `parse_smpte_timecode` 把 SMPTE 帧解回毫秒。结果: 10000ms 字幕 → 序列化变 250 → 重解析变 ~250ms。

**修复**: 直接存毫秒, 与 `parse_smpte_timecode` 输出契约对齐。

```rust
// ebu_stl.rs::to_string (line 388+)
let tti_blocks: Vec<TtiBlock> = subtitles
  .iter()
  .enumerate()
  .map(|(i, sub)| {
    TtiBlock {
      subtitle_group: 0,
      subtitle_number: (i + 1) as u16,
      extension_block: 0,
      cumulative_status: 0,
      timecode_start: sub.start as u32,   // 直接存毫秒, 删除 / 40
      timecode_end: sub.end as u32,       // 直接存毫秒, 删除 / 40
      // ... 其余字段不变
    }
  })
  .collect();
```

`model/builder.rs:183-184` 同款 `/40` 一起修 (改为直接 `as u32`)。

`encode_smpte_timecode` 与 `parse_smpte_timecode` 毫秒契约不变 (已经是对的)。

**regression 测试**: `test_ebu_stl_round_trip` — 构造几个 Subtitle, 序列化后 reparse, `start`/`end` 误差 < 80ms (SMPTE 25fps 一帧 = 40ms, 2 帧容差)。再加 `test_ebu_stl_round_trip_multiple` 覆盖批量场景。

### 2.2 Fix 2: SCC drop-frame (完整 SMPTE 12M-1-2014)

**根因**: `scc_timecode_to_ms` (scc.rs:172-176) 两个问题:
1. `fps as u64` 把 29.97 截成 29 — fps 应全程 f64。
2. drop-frame 与非 drop-frame 用同一公式, NTSC 长视频每小时漂 3.6s。

**修复**: 标准 SMPTE 12M-1-2014 §3.3 算法。

```rust
/// SCC/CEA-608 timecode → milliseconds.
///
/// Implements SMPTE 12M-1-2014 §3.3 drop-frame algorithm for NTSC
/// (29.97 fps). Non-drop timecodes use simple frame×(1000/fps).
fn scc_timecode_to_ms(h: u64, m: u64, s: u64, f: u64, fps: f64, drop_frame: bool) -> u64 {
  // 用整数帧避免 f64 精度损失; 30 是 NTSC 标称帧率 (实际 29.97)
  let nominal_fps = fps.round() as u64;   // 29.97 → 30
  let mut total_frames = h * 3600 * nominal_fps
    + m * 60 * nominal_fps
    + s * nominal_fps
    + f;

  if drop_frame {
    // SMPTE 12M-1-2014 §3.3: 每分钟丢 2 帧 (帧序号 0,1 跳过), 但每整 10 分钟不丢。
    // drop_count = 2 * (total_minutes - total_minutes / 10)
    //   - total_minutes = 0  → drop 0  (00:00:00;00 起点不丢)
    //   - total_minutes = 1  → drop 2  (00:01:00;00 时已丢 2 帧)
    //   - total_minutes = 9  → drop 18
    //   - total_minutes = 10 → drop 18 (10 分钟这一分钟不额外丢, 仍累计 18)
    //   - total_minutes = 60 → drop 108 (1 小时整)
    let total_minutes = h * 60 + m;
    let drop_count = 2 * (total_minutes - total_minutes / 10);
    total_frames -= drop_count;
  }

  // 帧数 → 毫秒: 用真实 fps (29.97), 不是标称 30
  ((total_frames as f64) / fps * 1000.0).round() as u64
}
```

`ms_to_scc_timecode` (scc.rs:179-200) 反向实现同样算法 — drop_frame 入参用于决定输出 `;` 还是 `:`, 但帧数反算也要应用 drop 补偿。

`scc.rs:80` 调用处把已有的 `drop_frame` 变量传进 `scc_timecode_to_ms` (现在签名加一个参数)。

**regression 测试** (预期值手算复核, 见下方"数值验证表"):
- `test_scc_nondrop_accuracy`: `01:00:00:00` (non-drop, fps=29.97) → 3603604ms
- `test_scc_dropframe_accuracy`: `01:00:00;00` (drop, fps=29.97) → 3600000ms
- `test_scc_dropframe_edge_zero_minute`: 验证多个时间点的丢帧数符合 SMPTE 12M
- `test_scc_round_trip`: parse → render → parse 时间码稳定

**数值验证表** (手算, 确保测试断言正确):

| 时码 | h,m,s,f | total_frames (前) | drop_count | total_frames (后) | ms = frames/29.97×1000 (round) |
|------|---------|-------------------|-----------|-------------------|-------------------------------|
| `00:00:00;00` (drop) | 0,0,0,0 | 0 | 0 | 0 | 0 |
| `00:01:00;00` (drop) | 0,1,0,0 | 1800 | 2 | 1798 | 59993 |
| `00:10:00;00` (drop) | 0,10,0,0 | 18000 | 18 | 17982 | 599800 |
| `00:11:00;00` (drop) | 0,11,0,0 | 19800 | 20 | 19780 | 659996 |
| `01:00:00;00` (drop) | 1,0,0,0 | 108000 | 108 | 107892 | 3600000 |
| `01:00:00:00` (non-drop) | 1,0,0,0 | 108000 | — | 108000 | 3603604 |

**关键不变量**:
- `01:00:00;00` (drop) = 3600000ms **正好 1 小时** — 这是 drop-frame 的设计目的 (时码与真实时间对齐)。
- `01:00:00:00` (non-drop) = 3603604ms — 多出 3604ms ≈ 108 帧 × 33.37ms。
- `00:10:00;00` 与 `00:11:00;00` 的差 = 659996 - 599800 = 60196ms ≈ 1 分钟 (而非 59993ms), 验证"整 10 分钟那一分钟不丢帧"。

### 2.3 Fix 3: encoding UTF-16 BOM

**根因**: encoding.rs:33-52 UTF-16 路径不剥 U+FEFF BOM (UTF-8 路径 line 31 剥了), `chunks_exact(2)` 静默丢奇字节尾部, 短输入无守卫。

**修复**: 合并 UTF-16BE/LE 单分支, 跳过 2 字节 BOM 后再 chunks, 结果 `trim_start_matches('\u{FEFF}')` 双保险, 短输入返回空串。

```rust
"UTF-16BE" | "UTF-16LE" => {
  if data.len() < 2 {
    return Ok(String::new());
  }
  let body = &data[2..];   // 跳过 2 字节 BOM (0xFEFF 编码后的 FF FE / FE FF)
  let u16: Vec<u16> = body
    .chunks_exact(2)
    .map(|c| {
      if encoding == "UTF-16BE" {
        u16::from_be_bytes([c[0], c[1]])
      } else {
        u16::from_le_bytes([c[0], c[1]])
      }
    })
    .collect();
  let s = String::from_utf16(&u16).map_err(|e| SubtitleError::InvalidEncoding {
    encoding: encoding.to_string(),
    error: format!("{:?}", e),
  })?;
  // 双保险: BOM 已被 body=&data[2..] 跳过, 这里再 trim 以防万一
  Ok(s.trim_start_matches('\u{FEFF}').to_string())
}
```

奇字节尾部由 `chunks_exact` 自动丢弃 — 在函数注释里说明这是有意行为 (UTF-16 严格 2 字节对齐)。

**regression 测试**:
- `test_decode_utf16be_bom_stripped`: 输出第一字符不是 `\u{FEFF}`
- `test_decode_utf16le_bom_stripped`: 同上 LE 变体
- `test_decode_utf16_odd_byte_ignored`: 5 字节输入不 panic, 结果含前 2 字符
- `test_decode_utf16_empty_input`: 0/1 字节输入返回空串, 不 panic

### 2.4 Fix 4: parse_to_file / cmd_parse 二进制误解码

**根因**: main.rs:97 (`parse_to_file`) 和 main.rs:135 (`cmd_parse`) 在 match 前 `decode_to_string(data)?`, EBU STL 二进制文件含任意字节触发 InvalidEncoding。

**修复**: 把 EbuStl 分支提到 match 之前早返回, 其余分支仍走文本解码。`parse_to_file`:

```rust
async fn parse_to_file(data: &[u8], format: CliFormat) -> AnyResult<SubtitleFile> {
  // 二进制格式跳过文本解码
  #[cfg(feature = "ebu_stl")]
  if let CliFormat::EbuStl = format {
    return Ok(ebu_stl::parse_content(data)?);
  }

  // 其余格式需要文本
  let text = subtitler::encoding::decode_to_string(data)?;
  match format {
    #[cfg(feature = "srt")]
    CliFormat::Srt => srt::parse_content(&text),
    // ... 11 个文本格式分支不变
    #[cfg(feature = "ebu_stl")]
    CliFormat::EbuStl => unreachable!("handled above"),
  }
}
```

`cmd_parse` 同样改造。SCC 是文本格式 (hex 编码), 仍走 `decode_to_string`。

**regression 测试**: `tests/cli_binary_format.rs` (新增集成测试) `test_parse_stl_skips_text_decode` — 构造含非法 UTF-8 字节的伪 STL 输入, 验证 `parse_to_file` 不报 `InvalidEncoding`。

### 2.6 Fix 6: split_long 零时长

**根因**: trait.rs:245 `chunk_duration = duration / num_chunks` 整数除法。`num_chunks > duration` (如 40ms 拆 5 段) → `chunk_duration = 0` → 中间段 `start == end` → `validate()` 报 ZeroDuration。

**修复**:

```rust
fn split_long(&mut self, max_chars: usize) {
  self.sort();
  let subs = self.subtitles_mut();

  let mut i = 0;
  while i < subs.len() {
    let char_count = subs[i].text.chars().count();
    if char_count <= max_chars {
      i += 1;
      continue;
    }

    let start = subs[i].start;
    let end = subs[i].end;
    // 守卫: 零或负时长字幕不拆
    if end <= start {
      i += 1;
      continue;
    }
    let duration = end - start;
    let style = subs[i].style.clone();
    let actor = subs[i].actor.clone();
    let text = std::mem::take(&mut subs[i].text);

    let chunks = split_text_chunks(&text, max_chars);
    let num_chunks = chunks.len() as u64;
    // .max(1) 保证每段非零时长 (末段吸收余数)
    let chunk_duration = (duration / num_chunks).max(1);

    // ... 其余 splice 逻辑不变
  }
}
```

**regression 测试**: `test_split_long_no_zero_duration` — 构造 50ms 字幕拆 10 段, 验证每段 `duration_ms() > 0` 且 `validate()` 无 ZeroDuration。

### 3.1 TDD 循环 (每个 Fix)

```
1. 先写 regression 测试 (能复现 bug)
   ↓ cargo test 失败 (确认 bug 存在)
2. 写最小修复
   ↓ cargo test 通过
3. 跑全套门禁 (clippy --all-targets / fmt / build / test / 最小构建)
   ↓ 全绿
4. 单独 commit
```

### 3.2 Fix 间回归保护

每完成一个 Fix 后, 除该 Fix 自己的测试, 还要重跑**之前 Fix 的 regression 测试**, 确保没破坏前面的修复。Fix 2/6 都动 `model/trait.rs` / `model/builder.rs` 附近代码, 特别注意。

### 3.3 最终验证门禁 (Task 8 之前)

```bash
cargo fmt -- --check
cargo clippy --all-targets -- -D warnings   # 必须带 --all-targets (AGENTS.md §3.1)
cargo test --all-targets                     # 测试数应从 273 增至 ~284 (11 新测试)
cargo build --no-default-features --features srt
cargo build --examples
cargo publish --registry crates-io --dry-run
git status                                   # 必须干净
```

### 3.4 失败回滚策略

- 单个 Fix 验证失败: `git revert HEAD`, 重新分析。**不在错误改动上硬修** (AGENTS.md §6.4)。
- 依赖升级出现 breaking: 保留该依赖原版本, CHANGELOG 注明"X crate 因 breaking 推迟到 2.2"。
- **SCC drop-frame 算法验证失败 (最可能失败的步骤)**: 回滚后回到 brainstorming 重新评估, 可能降级为 best-effort + 文档说明。前面 5 项修复已 commit 不受影响, 可发 2.1.0 不含 SCC 修复, 发 2.1.1 补。

### 6.1 MIGRATION.md 增补草稿

## 2.0 → 2.1

### 行为变更

- **SCC drop-frame 时间码**: 之前 SCC 用非 drop-frame 公式处理所有时码,
  导致 NTSC 长视频 (drop-frame 分隔符 `;`) 每小时漂移约 3.6 秒。2.1
  正确实现 SMPTE 12M drop-frame 算法。如果你之前依赖错误的时码值,
  请重新解析你的 SCC 文件。

- **EBU STL 序列化**: 之前 `ebu_stl::to_string` 写出的 STL 文件 timecode
  字段是损坏的 (写入 `ms / 40` 而非真实毫秒)。2.1 修复后写出的 STL 文件
  可被正确 round-trip 解析。旧错文件无法被新版解析 —— 但这些文件本身
  不可用, 不太可能有依赖。

- **UTF-16 解码**: UTF-16 编码的字幕文件现在正确剥离 U+FEFF BOM (与
  UTF-8 路径行为一致)。如果你的下游代码依赖 BOM 前缀, 需要相应调整。

### 新增

- `encoding::decode_to_string` 现在对 UTF-16 输入跳过 2 字节 BOM 后再
  解码, 短输入 (≤1 字节) 返回空串而非 panic。
