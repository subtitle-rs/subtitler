## Code Fixes

### Task 1: Fix clippy needless borrow in lrc.rs

**Files:**
- Modify: `src/lrc.rs:268`

- [ ] **Step 1: Apply the fix**

In `src/lrc.rs`, inside `mod tests`, the `test_round_trip` function at line 268 has:

```rust
    let output = to_string(&subs);
```

Change to:

```rust
    let output = to_string(subs);
```

Rationale: `subs` is `&[Subtitle]` (returned by `file.subtitles()`), and `to_string` takes `&[Subtitle]`. The `&` creates a `&&[Subtitle]` that the compiler auto-derefs — clippy flags this as `needless_borrow`.

- [ ] **Step 2: Verify clippy now passes for lrc**

Run:
```bash
cargo clippy --all-targets -- -D warnings 2>&1 | tail -10
```
Expected: `Finished` with no errors. If a NEW clippy error appears elsewhere, do not fix it here — note it for the assignee but continue (this plan only covers the 2.0.1 scope).

- [ ] **Step 3: Verify the test still passes**

Run:
```bash
cargo test --lib lrc 2>&1 | tail -5
```
Expected: `test result: ok.` with all lrc tests passing.

- [ ] **Step 4: Commit**

```bash
git add src/lrc.rs
git commit -m "fix(lrc): remove needless borrow flagged by clippy

Unblocks 'cargo clippy -- -D warnings' CI gate that was failing
on the 2.0.0 release. The &subs in test_round_trip created an
unnecessary &&[Subtitle] that the compiler auto-derefs."
```

2. **Tag the release** (user decision):
   ```bash
   git tag -a v2.0.1 -m "v2.0.1: hotfix — clippy fix, CHANGELOG/README/Cargo.toml corrections"
   git push origin main --tags
   ```

3. **Publish to crates.io** (user decision):
   ```bash
   cargo publish --dry-run   # sanity check
   cargo publish             # actual publish
   ```

4. **Optionally create a GitHub Release** — the `release.yml` workflow handles this on tag push if configured; otherwise create manually from the tag, pasting the `[2.0.1]` CHANGELOG section as the release notes.

## Rollback

If any verification in Task 12 fails:

- **Code fix failures (Tasks 1-2)**: `git revert <commit>` and re-investigate. Do not paper over clippy failures.
- **Docs failures (Tasks 4-11)**: usually safe to amend in place; the verification gate in Task 12 catches regressions.
- **If already tagged/published**: this is a patch release, so forward-fix in `2.0.2` rather than yanking `2.0.1` (yanking has its own costs and should be reserved for security/correctness regressions, which this hotfix cannot introduce).
