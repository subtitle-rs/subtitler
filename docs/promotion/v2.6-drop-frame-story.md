# SMPTE 12M Drop-Frame 时间码：我用 Python 证明自己算错了

> 实现 SCC 格式的 drop-frame 时间码时，我手算的测试预期值错了两次。
> 最终用 Python 模拟才发现：drop-frame 的设计目的和我想的不一样。

## 背景：SCC 字幕的时间码问题

SCC（Scenarist Closed Caption）是北美广播电视的字幕格式，用 SMPTE 时间码 `HH:MM:SS;FF`（注意分号——这代表 drop-frame）。

NTSC 电视的帧率是 29.97 fps，不是整数 30。如果用非 drop-frame，显示时间码 1 小时时实际已经过了 1 小时 3.6 秒。SMPTE 12M 标准的解法是**每分钟丢 2 帧（帧号跳过 0 和 1），每 10 分钟不丢**。

## 算法不复杂

```rust
fn scc_timecode_to_ms(h: u64, m: u64, s: u64, f: u64, fps: f64, drop_frame: bool) -> u64 {
    let nominal_fps = fps.round() as u64;  // 29.97 → 30
    let mut total_frames = h * 3600 * nominal_fps
                         + m * 60 * nominal_fps
                         + s * nominal_fps
                         + f;
    if drop_frame {
        let total_minutes = h * 60 + m;
        let drop_count = 2 * (total_minutes - total_minutes / 10);
        total_frames -= drop_count;
    }
    ((total_frames as f64) / fps * 1000.0).round() as u64
}
```

## 但预期值算错了

我写 spec 时手算 `00:10:00;00` (drop)：

```
total_frames = 10 × 60 × 30 = 18000
drop_count = 2 × (10 - 1) = 18
net = 18000 - 18 = 17982
ms = 17982 / 29.97 × 1000 = 599800
```

测试跑出来：**600000**。差了 200ms。

## Python 救场

```python
def to_ms(h, m, s, f, fps=29.97, drop=False):
    nf = round(fps)  # 30
    tf = h*3600*nf + m*60*nf + s*nf + f
    if drop:
        tm = h*60 + m
        tf -= 2*(tm - tm//10)
    return round(tf / fps * 1000)

print(to_ms(0, 10, 0, 0, drop=True))  # 600000 ???
```

Python 也给 600000。我手算哪里错了？

```
17982 / 29.97 = ???
```

**29.97 × 600 = 17982.0**——整除！`17982 / 29.97 × 1000 = 600000.0`。

我手算时把 `17982 / 29.97` 当成 `~599.8`，实际是 `600.0`。

## Drop-frame 的真正设计目的

这才是关键发现。我原以为 drop-frame 只是"补偿帧率差异"，但它有精确的不变量：

| 时间码 | 真实时间 | 含义 |
|--------|---------|------|
| `01:00:00;00` (drop) | **3600000ms** | 正好 1 小整 |
| `01:00:00:00` (non-drop) | 3603604ms | 多 3.6 秒 |

**Drop-frame 的设计目的就是让整 10 分钟和整小时的时间码与真实经过时间精确对齐。** `00:10:00;00` = 正好 10 分钟，`01:00:00;00` = 正好 1 小整。

这不是巧合——`29.97 × 3600 = 107892`，而 drop-frame 让 `01:00:00;00` 对应的帧数正好是 107892。数学是自洽的。

## 教训

1. **纸面推导会错**——尤其是浮点除法和四舍五入的组合
2. **先怀疑预期值，再怀疑实现**——TDD 测试失败时两种都查
3. **Python 是最快的草稿验证工具**——比纸算可靠，比 Rust 单测快
4. **理解设计意图比记公式重要**——知道"为什么 drop-frame 存在"才能正确判断结果对不对
