# 从 2000 次堆分配到 0：subtitler 的零拷贝解析实践

> "零拷贝"听起来像是要用 mmap、arena allocator 或者给每个结构体加生命周期参数。
> 但 subtitler v2.0 做的零拷贝没那么花哨——它只做了一件事：**停止做不需要做的事情**。

## 问题：1000 条字幕 = 2000 次 malloc

SRT 文件每行大概长这样：

```
1
00:00:01,000 --> 00:00:03,000
Hello world
```

v1.0 的解析器对每一行都做了：

```rust
let mut trimmed = line.trim().to_string();
```

`line` 来自 `content.lines()` 迭代器，是 `&str`——直接指向输入内存，零分配。`.trim()` 返回更小的 `&str`——还是零分配。然后 `.to_string()` 复制到一个新的堆分配 `String` 里。

用完就丢。下一行又来一次。

1000 条字幕 ≈ 2000 行 → **2000 次 `malloc` + 2000 次 `free`**，只为了得到一些用完就丢的临时字符串。

## 修复：一行之差

```rust
// v1.0
let mut trimmed = line.trim().to_string();  // 分配

// v2.0
let trimmed = line.trim();  // 零分配
```

现在 `trimmed` 是 `&str`。检查空行？`trimmed.is_empty()`。解析序号？`trimmed.parse::<usize>()`。追加到字幕文本？`sub.text.push_str(trimmed)`——这里仍然需要拷贝，因为字幕数据总要存储，但这个拷贝是**必须的**，而不是**多余的**。

**本质**：把解析过程中产生的临时字符串全部消除，只保留最终进入 `Subtitle` 结构体的那一次拷贝。

## Vec 预分配：消灭 realloc 偷跑

v1.0 的解析器从 `Vec::new()` 开始（capacity=0）。第一条字幕 push 时分配 4 个元素空间，第 5 条时 realloc 到 8，第 9 条到 16……

v2.0 在第一行就知道大概有多少条字幕：

```rust
// 每条字幕大约 200 字节
let estimated = (content.len() / 200).max(16);
let mut subtitles: Vec<Subtitle> = Vec::with_capacity(estimated);
```

EBU STL 更精确——直接从 GSI 头读取 TTI 数量：

```rust
let tti_count = u16::from_be_bytes([data[1], data[2]]) as usize;
let mut subtitles = Vec::with_capacity(tti_count);  // 0 次 realloc
```

## split_text_chunks：从 O(n²) 到 O(n)

`split_long()` 把长字幕按词拆分。v1.0 用 `format!("{} {}", current, word)` 每次重建整个字符串——经典的 Shlemiel the painter 问题。

v2.0 改为 `push_str` 追加：

```rust
let mut current = String::with_capacity(max_chars);
// ...
current.push(' ');
current.push_str(word);
```

`format!()` 调用从 99 次降到 0 次。

## 为什么不改成 `Subtitle<'a>`？

理想但破坏性的方案：

```rust
pub struct Subtitle<'a> {
    pub text: &'a str,  // 直接借用输入
}
```

评估后拒绝了，四个原因：

1. **传染性**：`Subtitle<'a>` → `SubtitleFile<'a>` → `SubtitleFormat<'a>` trait → 所有方法签名 → CLI → `main()`。50+ 文件、47 个编译错误。
2. **merge_adjacent 困难**：合并两段文本需要拼接，`&str` 无法修改。
3. **`into_static()` 惩罚**：任何离开解析上下文的操作都要克隆，零拷贝省的在边界还回去。
4. **用户代码破坏**：每个持有 `SubtitleFile` 的类型都要加生命周期。

**最佳折衷**：内部零拷贝（解析路径无临时 String）+ 边界必需拷贝（Subtitle 自己拥有数据）。

## 效果

| 指标 | v1.0 | v2.0+ |
|------|------|-------|
| 临时 String 分配 | ~100,000 | **0** |
| Vec realloc | ~15 | **0-1** |
| `split_text_chunks` | O(n²) | **O(n)** |
| API 破坏 | — | **0** |

不需要 `unsafe`、不需要自定义分配器、不需要改变公开 API。只需要对 Rust 字符串和 Vec 的底层机制有一点理解：

- `str::trim()` 返回 `&str`，不分配
- `Vec::with_capacity` 一开始就知道要多大
- `String::push_str` 追加比 `format!()` 重建快一个数量级

**零拷贝不是魔法——它是"不做不需要做的事"重讲了一遍。**
